## 3. 가장 어려웠던 것 TOP 7

### ① 브라우저는 "자기 탭 안"만 봐요

**문제**  
D-Garden 탭을 켜 두고 Cursor에서 코딩하면, 브라우저 입장에서는 입력이 없으니까 **idle(휴면)** 로 떨어져요. 데모의 핵심인 "타이핑하면 정원이 반응한다"가 그대로 깨져요.

**시도 1 — Cursor/VS Code 확장 (`extension/`)**  
에디터 이벤트를 로컬 SSE(`127.0.0.1:7331`)로 보내는 확장을 만들었어요. 동작은 했는데 **Cursor 전용**이라 범위가 좁았어요.

**시도 2 — 전역 입력 에이전트 (`agent/`)**  
`uiohook-napi`로 OS 전체 키/마우스 **횟수만** 후킹하는 백그라운드 프로세스로 바꿨어요. 키 내용은 절대 전송하지 않아요.

**팁**
- 웹앱 + "실제 작업 중 입력"을 연결하려면 **브라우저 밖 브리지**가 필요해요.
- 프라이버시 스토리("키 내용 미수집")를 제품 신뢰의 핵심으로 박아 두면, 사내 웰니스로 확장할 때 유리해요.
- 포트 충돌(`EADDRINUSE`)이 자주 나요. Cursor 확장 uninstall → Cursor 재시작 → 에이전트 실행 순서를 문서로 남겨 두면 좋아요.

### ② Canvas 생성형 아트 — "예쁘게"와 "안정적으로"가 부딪혀요

**문제**  
프랙탈 나무, 3중 주파수 바람, 스프링 물리, 잎 생장 체인… AI가 코드를 잘 짜줘도 **눈으로 보는 미세 감각**은 기획자가 직접 잡아야 했어요.

**대표 사례 — OPTIMAL 상태의 "나비효과"**  
6/12 저녁에 잎 flutter·바람·가지 sway 관련 fix/perf 커밋만 10개 이상 쌓였어요.

1. 잎이 점점 빨라져 보임 (beating — 여러 주파수 sine 파형이 간섭하는 현상)
2. cuteBoost로 OPTIMAL을 귀엽게 만들려다 오히려 산만해짐
3. 결국 `windPower=0`, 단일 느린 sine 파형, sway 97% 감소로 **"최적 몰입 = 절대 정적"** 에 수렴했어요

**배운 점**
- 생성형 아트는 **"더 움직이게"가 아니라 "언제 멈출지"** 가 더 어려워요.
- 수치는 `spec.json`에 두되, **시각 수용 기준**을 말로라도 정해 두세요. ("OPTIMAL에서는 잎이 거의 안 움직인다")
- AI에게 "느리게 해줘"만 반복하면 진동·간섭 같은 **물리 버그**가 새로 생겨요. 원인(다중 주파수, freqMult 증폭)을 짚어 주는 게 훨씬 빨라요.

**다른 그래픽 버그들**
- `spawnSeed` 인자 순서 오류 → 색 깨진 거대 씨앗이 무한 생성
- 글로잉나무 잎이 연꽃처럼 보임 → 분기점 방사형 3장 잎을 말단 가지로 이동
- 비네트·앰비언트가 너무 어두워 글래스 UI가 죽음 → `FX.VIGNETTE_ALPHA`, `FX.AMBIENT_ALPHA`로 spec에서 감량

### ③ SDD/TDD를 지키다가, 막판에 깨뜨렸어요

**잘 했던 것**
- `.cursorrules`에 "수치 하드코딩 금지, 테스트 먼저" 명시
- `spec.json`에 점수·CPM·키워드·팔레트·FX 전부 명세
- `garden.test.js` 33개 → `agent/keycount` 추가로 49개

**깨진 것 (발표 직전 데모 튜닝)**
- `5ff9332`: 점수 민감도 완화 (idle -0.5→-0.2, burst 1.0→0.6 등)
- `328020c`: CPM 임계값 상향 (FOCUS 300→500, OVERHEAT 400→700)
- **`garden.test.js`는 PRD 원본 수치 기대값 그대로** → 10개 실패

**가장 중요한 팁**  
`spec.json`을 바꿀 때는 **반드시 테스트도 같이** 바꿔 주세요. 더 좋은 방법은 테스트가 spec 값을 읽어 기대값을 계산하게 만드는 거예요:

```javascript
// 나쁜 예: 하드코딩 기대값
expect(calcIdleDecay(50)).toBeCloseTo(49.5);

// 좋은 예: spec 기반
expect(calcIdleDecay(50)).toBeCloseTo(50 + SPEC.IDLE_DECREASE_PER_SEC);
```

비개발자가 AI를 "통제"한다는 스토리의 핵심은 **코드를 읽는 것**이 아니라 **테스트 GREEN/RED를 보는 것**이에요. 이걸 끝까지 지키지 못한 게 가장 아쉬운 실수였어요.

### ④ 관객 인터랙션 — 키워드 매칭의 함정

**문제**  
6버튼 리모컨("오~", "와우", "굿", "에이", "별로", "흠...")을 눌러도 점수가 안 변했어요.

**원인**  
`spec.json`의 `POSITIVE_KEYWORDS` / `NEGATIVE_KEYWORDS`에 버튼 텍스트가 없었고, 문자열 매칭만 의존하고 있었어요.

**해결**
- 버튼 텍스트를 키워드에 추가
- `onCrowdDirect`로 sentiment를 직접 전달 (키워드 우회)
- CPM burst 트리거 임계값 상향으로 관객 입력 시 과열 오작동 방지

**팁**
- 해커톤 데모는 **자유 텍스트 입력보다 고정 버튼**이 훨씬 안정적이에요.
- Realtime은 **Broadcast만** 쓰면 DB 스키마·RLS 걱정 없이 빨리 붙일 수 있어요. 기존 Supabase 프로젝트 재활용이 정답이었어요.

### ⑤ 발표 연출과 제품 연동

**연출**  
화면 **좌측 절반 = 슬라이드**, **우측 절반 = 라이브 D-Garden**. 슬라이드 첫 장은 QR 온보딩이에요.

**어려웠던 점**
- 슬라이드는 `presentation/slides.html`로 별도 제작 — 본편 UI 토큰(글래스, Space Mono, 네온)을 맞춰야 통일감이 나요
- 관객이 QR로 미리 들어와 있어야 발표 중 반응이 살아나요 → **리허설 필수**인데 끝까지 못 했어요

**팁**
- 해커톤 발표는 **"보여주기 좋은 것"** 과 **"돌아가는 것"** 을 분리해서 준비해 보세요.
- 슬라이드에 "지금 눌러보세요" 한 장을 넣고, 우측 라이브 화면을 가리키는 구조가 효과적이에요.

### ⑦ 비개발자 + AI 협업의 실제 난이도

**쉬웠던 것**
- "프랙탈 나무 + 물리 바람 + 4테마" 같은 큰 덩어리를 AI가 한 번에 구현
- 그래픽 리뷰 피드백을 자연어로 넘기면 일괄 반영
- Supabase/Vercel 같은 인프라 보일러플레이트

**어려웠던 것**
- **버그 원인 진단** — "왜 잎이 연꽃처럼 보이지?" "왜 점수가 안 오르지?"는 증상 → 원인 추적이 필요해요
- **수치 튜닝** — 감각적 판단(너무 빠르다, 너무 어둡다)은 사람만 할 수 있어요
- **범위 통제** — AI가 관련 없는 파일까지 건드릴 때 `.cursorrules`와 "이 파일만" 지시가 필수예요
- **검증** — `npm test`, 브라우저에서 직접 눌러보기, 폰으로 QR 테스트

**팁 — AI에게 시키기 좋은 것 vs 사람이 할 것**

| AI에게 맡기기 좋음 | 사람이 직접 할 것 |
|-------------------|------------------|
| 보일러플레이트·반복 구현 | 제품 방향·우선순위 결정 |
| 테스트 케이스 작성 | 테스트 GREEN 확인 (비개발자도 가능해요) |
| spec.json 수치 반영 | "이 정도면 됐다" 시각 판단 |
| 버그 수정 (원인 알려주면) | 데모 시나리오·발표 스크립트 |
| 문서·슬라이드 초안 | 관객 리허설 |

## 4. 아키텍처 결정과 이유

```
[사용자 PC]
  ├─ agent/index.js (선택) ──SSE──► engine/editor-bridge.js
  ├─ index.html (메인 가든)
  │    ├─ engine/tracker.js    ← 키보드·마우스·crowd 집계
  │    ├─ garden.js            ← 순수 점수 로직 (TDD)
  │    ├─ engine/renderer.js   ← Canvas 생성형 아트
  │    └─ engine/realtime.js   ← Supabase Broadcast 수신
  └─ audience.html (관객 리모컨) ──Broadcast──► realtime.js

[spec.json] ← 모든 수치·키워드·팔레트·FX (SDD 단일 소스)
```

| 결정 | 이유 |
|------|------|
| Vanilla JS (프레임워크 0) | 빌드 없이 Vercel 정적 배포, AI가 파일 단위로 수정하기 쉬워요 |
| LLM API 0 (런타임) | 멘탈 스코어는 CPM+오타율 경량 알고리즘으로 충분하고, 비용·지연도 없어요 |
| spec.json 단일 명세 | 비개발자가 수치 튜닝할 수 있고, AI 하드코딩도 막을 수 있어요 |
| Supabase Broadcast only | DB 없이 실시간 연동, 기존 프로젝트 재활용도 안전해요 |
| 4테마 + localStorage | 데모 때 심사위원에게 "골라보세요" 할 수 있어요 |
| 전역 에이전트 (확장 only 아님) | Cursor 밖 타이핑도 반영돼서 데모 신뢰도가 올라가요 |
